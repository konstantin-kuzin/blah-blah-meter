# Time Shit — документация проекта

Клиентское веб-приложение для оценки **стоимости встречи** по составу участников (грейды и оклады) и **прошедшему времени**. Поверх основного интерфейса доступна **транскрипция речи** (браузер или Groq Whisper через серверный прокси).

---

## 1. Способ реализации

### 1.1. Общая архитектура

- **Фронтенд:** одностраничное приложение без сборщика — `index.html`, `styles.css`, `app.js`, `speech.js`.
- **Состояние:** настройки встречи и речи хранятся в `localStorage` под ключом `startTracker.settings.v1`; тема — `startTracker.theme.v1`.
- **Серверная часть (опционально):** каталог `api/` — функция для платформ в стиле Vercel Serverless. Маршрут **`/api/groq`** проксирует запросы к API Groq (транскрипция и чат), чтобы **не светить ключ в публичном репозитории** и использовать переменную окружения `GROQ_API_KEY`.
- **Локальная разработка с Groq:** скрипт `start.cjs` поднимает HTTP-сервер со статикой и тем же обработчиком `api/groq.js`, что и на Vercel (см. комментарии в файле).

### 1.2. Расчёт стоимости

Для каждого грейда минутная ставка:

```text
ratePerMinute = (salary × 1.5) / (160 × 60)
```

- `salary` — месячный оклад (руб.);
- `1.5` — грубый коэффициент на налоги и взносы;
- `160` — рабочих часов в месяце.

Итоговая стоимость встречи — сумма ставок всех выбранных участников, умноженная на длительность встречи. Обновление отображения суммы делается с интервалами ~250 ms (руб.) и ~62.5 ms (копейки) для плавности.

Встреча может дополнительно разбиваться на **сегменты (раунды)**:

- первый сегмент стартует одновременно с началом встречи;
- кнопка `Раунд` завершает текущий сегмент и сразу запускает следующий;
- в ленте под таймером отображаются только завершённые сегменты;
- для каждого завершённого сегмента показываются его длительность и стоимость;
- стоимость активного сегмента считается в реальном времени и выводится на кнопке `Раунд`;
- при изменении времени начала встречи или полном сбросе встречи сегменты пересоздаются заново.

### 1.3. Модульность скриптов

- **`app.js`** — загрузка/нормализация настроек, рендер карточек грейдов, таймер встречи, учёт сегментов встречи, модальное окно настроек (вкладки «Участники» / «Speech Recognition»), сохранение формы, интеграция с `startTrackerSpeech.configure()` после сохранения.
- **`speech.js`** — изолированный модуль, экспортируемый в глобальный объект `startTrackerSpeech` (`init`, `configure`, `handleDocumentEscape`).

### 1.4. Развёртывание

- **Только статика:** достаточно открыть `index.html` или отдать файлы любым статическим хостингом — расчёт встречи и Web Speech API работают без бэкенда.
- **Vercel + Groq:** задеплоить репозиторий, задать `GROQ_API_KEY` в настройках проекта; клиент ходит на относительный путь `/api/groq`.
- **Альтернатива ключу на сервере:** пользователь может ввести ключ в настройках; он уходит в заголовке `Authorization: Bearer …` при вызове `/api/groq` (ключ на стороне клиента — компромисс удобства и безопасности).

## 2. Основные функции взаимодействия

| Область | Действие пользователя | Поведение |
|--------|------------------------|-----------|
| **Участники** | Кнопки `+` / `−` у каждого грейда | Меняется число участников данной должности; пересчёт стоимости и «руб/мин». |
| **Таймер** | Первое увеличение числа участников | Старт отсчёта длительности встречи (стена часов). |
| **Продолжительность** | Клик по «Продолжительность встречи» | Открывается попап с полем времени начала; после сохранения пересчитывается прошедшее время. |
| **Сегменты** | Кнопка `Раунд` под таймером | Закрывает текущий сегмент встречи и сразу запускает следующий; на кнопке показывается стоимость активного сегмента. |
| **История сегментов** | Лента карточек под таймером | Показывает завершённые сегменты встречи с их длительностью и стоимостью. |
| **Настройки** | Иконка шестерёнки | Модальное окно: заголовок страницы, список грейдов (название, оклад, добавить/удалить), вкладка настроек речи. |
| **Тема** | Иконка солнца/луны | Переключение `data-theme` на `<html>`, сохранение в `localStorage`. |
| **Микрофон** | Кнопка микрофона в тулбаре | Вкл/выкл распознавание (режим задаётся в настройках). **Shift+клик** при выключенном микрофоне — очистка текста транскрипта. |
| **Транскрипт** | Слой поверх фона (`aria-live`) | Финальный и промежуточный текст; прокрутка к концу при обновлении. |
| **Клавиатура** | Escape (при активном прослушивании) | Остановка захвата/распознавания (`handleDocumentEscape`). |

Настройки формы при «Сохранить» сериализуются в объект `settings` (включая `speech`) и записываются в `localStorage`; представление (заголовок, грейды, счётчики) обновляется сразу.

## 3. Распознавание речи

### 3.1. Два режима

1. **Web Speech API (браузер)**  
   - Используется `SpeechRecognition` / `webkitSpeechRecognition`.  
   - Язык распознавания в коде: `ru-RU`.  
   - Режим непрерывный (`continuous`), с промежуточными результатами (`interimResults`).  
   - Качество и доступность зависят от браузера и ОС.

2. **Groq Whisper**  
   - Звук с микрофона пишется через **`MediaRecorder`**, нарезается на фрагменты по времени (`GROQ_SLICE_MS`, по умолчанию 6 с).  
   - Каждый фрагмент кодируется в Base64 и отправляется на **`POST /api/groq`** с `action: "transcribe"` (сервер собирает multipart и вызывает Groq OpenAI-совместимый endpoint транскрипции).  
   - Очередь чанков обрабатывается последовательно (`groqQueue` + `pumpGroqQueue`), чтобы не перегружать API.  
   - Для микрофона в режиме Groq настраиваются ограничения захвата (в т.ч. отключение эхоподавления по умолчанию — см. комментарии в `speech.js`), чтобы уменьшить глушение звука из колонок в Chrome.

### 3.2. Постобработка (Groq Chat)

Если в настройках включена **постобработка в LLM**, после паузы в наборе текста (`POLISH_DEBOUNCE_MS`) отправляется запрос `action: "chat"` с системным промптом-«редактором» расшифровки (орфография, пунктуация, сохранение английских терминов). Результат заменяет накопленный транскрипт. При ошибке пользователь видит подсказку в `title` кнопки микрофона; исходный текст сохраняется.

### 3.3. Перевод (работает нестабильно, скрыто в настройках)

При включённом **переводе** в интерфейсе показывается не сырой транскрипт, а результат отдельного вызова `action: "chat"` с промптом перевода на English или 简体中文. Запросы **дебаунсятся** (`TRANSLATE_DEBOUNCE_MS`), есть учёт **429** (backoff), чтобы не упираться в лимиты RPM Groq. Постобработка и перевод **делят одну квоту** — в коде это явно учтено задержками и очередью.

### 3.4. Защита от мусора STT

На клиенте фильтруются типичные «галлюцинации» тишины (часть правил в `looksLikeSilenceHallucination`). На сервере для ответов `verbose_json` дополнительно отбрасываются сегменты с плохими метаданными (`no_speech_prob`, `compression_ratio`) — см. `api/groq.js`.

### 3.5. Отладка

В `localStorage` ключ `startTracker.speechDebug = "1"` включает подробные логи в консоли (`speech.js`).

### 3.6. Конфигурация в настройках

В объект `speech` входят, в частности:

- `mode`: `"browser"` | `"groq"`;
- `groqPostProcess`, `groqClientKey`;
- `sttModel`, `chatModel`;
- `translateEnabled`, `translateTarget` (`"en"` | `"zh"`).

Часть полей в UI показывается условно (модель транскрипции только для Groq, модель постобработки — при включённой постобработке, язык — при включённом переводе); логика в `app.js` (`updateSpeechSettingsFieldsVisibility`).

## 5. Ограничения и замечания

- Web Speech API и качество микрофона зависят от браузера; не все движки одинаково хорошо держат русский + вкрапления английского.
- Режим Groq требует **HTTPS или localhost** для `getUserMedia` в типичных браузерах; на Vercel это выполняется.
- Ключ Groq в поле настроек хранится вместе с остальными настройками в `localStorage` — для личного использования приемлемо; для продакшена предпочтительнее только серверный `GROQ_API_KEY`.
